Failure Pattern 为什么这会危险

黑盒不是“看不见思考”，而是看不见状态边界

文件链爆炸读了 5 个文件仍然想继续追旧日志，任务入口越来越不清楚。
权威混乱record、latest、clean recheck 混在一起，分数越多越难判断。
上下文继承错觉新窗口把旧窗口 token 当成当前状态，开始就带着误解。
完成声明漂移没有 evidence，也会说“可能完成了”。
修复越修越散benchmark 反馈变成不断加补丁，而不是一次最小进化。
工具越来越重越不确定，越想调用更重的工具，成本和风险一起上升。

Core Promise 不是救火工具，是进化引擎

你的 Agent 没崩溃，也可以继续进化

输入可以很普通
输入 A: 一次 BotLearn 跑分报告
输入 B: 一段工作日志 worklog
输入 C: 一次任务输出
输入 D: 一段失败/超时/漂移日志

不需要等它崩溃。
只要有一次运行，就可以提炼下一轮进化。
Evolution Card识别本轮最该提升的能力维度：perceive / act / memory / guard / autonomy / reason。
Minimal Skill Patch只给一个最小补丁，例如补 webhook 骨架、memory ledger、source coverage。
Promotion Gate只有通过 dry-run、证据审计和 receipt，才把补丁晋升为长期 skill。

它的招牌是：Agent 每做一次事，就多一个可复用的能力沉淀。

How It Works 失败变成最小进化

每一次失败，都留下下一轮可执行的修复路径

1
记录保存目标、证据、分数、上下文压力。
2
分层区分 trusted_state、uncertain_state、stale_claim。
3
停止超过 70% 不扩展，读满 5 个文件仍不清楚就停。
4
修复只生成一个最小 skill 补丁，不盲目重构。
5
验证dry-run、payload audit、receipt，再决定是否晋升。
For Audience 现场 30 秒体验

给它任意一次 Agent 记录，它输出四件能直接用的东西

输入：任意 Agent 运行记录
我的 Agent 刚跑完一轮 benchmark。
或者刚完成一个真实任务。
或者只有一段普通 worklog。

我想知道：
- 它哪项能力最值得提升？
- 哪些结论可信，哪些只是推断？
- 下一步最小 skill 补丁是什么？
- 怎么写成 field note 给别人看？
1. Snapshot提取当前状态、证据、未知点和停止条件。
2. Evolution Card告诉你这一轮应该提升哪个能力维度，以及为什么。
3. Skill Patch生成一个最小 skill 补丁或答题范式，不扩大系统复杂度。
4. Field Note自动生成可传播的测试记录：输入、变化、证据、下一步。

观众不需要先理解 SACP。只要给一段运行记录，就能看到自己的 Agent 如何从“完成一次任务”变成“沉淀一个能力”。

Closing 一句话带走
我们不是让 Agent 只完成一次任务。
我们让它每一次任务后，都变得更强。

Audit Evolution: 状态可恢复，过程可审计，经验可沉淀，Agent 可进化。
